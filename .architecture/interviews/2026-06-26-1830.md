## Q1 — What was the friction?

> So the moment of friction is that when this particular plug in is installed for the very first time that it is running a project, there is no structural architecture to support the plug in. Ideally, this would just be completed as part of the plug in installation, but in accordance to the way Claude code and co work function, there is an option for and needs to be an option for user permission and consent to make file directory changes. If this consent is automatic in Claude Co work and Claude code when file directory and structure changes happen, then there may not be a need for a full UX UI interview at this stage. It might just be that Co work and Claude code will say, I need permission to create dot architecture or to create the databases, etcetera, etcetera. But the onboarding should include these steps specifically rather than it being ad hoc and put on co work or put on Claude... or sorry, Claude code. This onboarding stages for a new project for this particular plug in should just be the same step by step onboarding so that it is Eventually expected by the user every time they use the plugin, they know what they need to say yes to, say no to, etcetera. Ultimately, this is a permissions unlock and consent of the user so that they know exactly what is being done to their system.

**Agent's pressure point:** Two answers in tension — "may not be a need for a full UX UI interview" AND "the onboarding should include these steps specifically." Needed to defend one.

## Q2 — Is the wizard supplementing per-write platform prompts, or replacing them?

> I think going with option b and having compass have its own onboarding sequence that includes these permissions and consents on mocks is the way to go.

**Selected:** B — Compass owns the onboarding sequence, replacing platform per-write prompts with a single explained walkthrough.

## Q3 — How do you avoid bundled consent becoming click-through theater?

> I agree that users won't pay attention, but we have given them the means to be informed and stay at the helm. That's the criteria. They could just say "I don't want an interview or to make decisions - just build it" and that's an informed release of control. That's what I'm going for. The default can't be that the agent is making decisions and taking actions - but the user should be allowed to opt-in to that.

**Resolved:** The wizard's purpose isn't to *prevent* click-through. It's to make click-through a *deliberate* act. "Just build it" is valid; default-to-passive-agent is not. Aligns with ADR 0006 (human-as-driver) and ADR 0008 (involvement is per-project, mutable).

## Q4 — Does the wizard re-run every session?

> i

**Selected:** (i) Wizard runs only when `.architecture/` is missing. Permission is per-project, persisted in the bootstrap output itself. Existence of `.architecture/` IS the consent record. "Just build it" gets persisted as the involvement setting via ADR 0008 within the project's own `.architecture/`.

## Q7 — Post-wizard UX: manual re-invoke or auto-pivot?

> I'm not sure why you're using different symbols for options each time. However, I think the best option here is β but with a twist. The onboarding runs, and then it reviews the last 10 passes to determine intent. That way it doesn't have to remember anything - there is just a trigger to review the active context and apply it.

**Selected:** β with a twist — auto-pivot via active-context review. No state-passing required; the agent already has the conversation in context. Heuristic: "last 10 turns" but the agent can ask "you wanted X, right?" if ambiguous before pivoting.

## Outputs

1. **Problem:** First-load UX — `compass:using-compass` invoked in a project without `.architecture/` currently *recommends* bootstrap, then proceeds passively. The recommendation gets ignored; subsequent skills degrade. Kevin's directive: assertive, consent-framed, single-host walkthrough.

2. **Load-bearing assumptions:**
   - Wizard owns onboarding (B from Q2); does not delegate to platform per-write prompts.
   - Frame is "informed-consent permission-unlock," not "feature-discovery onboarding."
   - Click-through is acceptable if it's a deliberate release of control; default-to-agent-takes-action is not acceptable.
   - Wizard runs only when `.architecture/` is missing (Q4-i). Existence of `.architecture/` is the consent record.
   - `compass:using-compass` is the sole host (Q5/Q7-A). Other skills error-and-redirect when invoked first.
   - Post-wizard, the agent reviews recent conversation context and resumes the user's original intent (Q7-β with twist).

3. **Open questions for premise-check:**
   - What's the literal wizard text? What options does it offer the user? Bootstrap-only? Bootstrap + involvement setting?
   - What happens if the user says "no" entirely (declines bootstrap)? Does the plugin become a no-op, print a warning on every skill, or what?
   - The "just build it" path: does it auto-pick a default involvement setting (e.g., hybrid), or does it skip the involvement-setting ADR creation entirely and let it default at first skill invocation?
   - Persistence of "just build it" — single ADR `0001-involvement-just-build-it.md` or a flag in manifest?
   - What does the "error-and-redirect" message in other skills literally say?

These open questions feed into the implementation plan.
